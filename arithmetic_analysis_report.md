# Анализ арифметических реализаций s21_decimal

## 1. Анализ ветки `feature/arithmetic/base`

### Архитектурные особенности
- **Модульная структура**: Каждая арифметическая операция реализована в отдельном файле
- **Единообразный API**: Все функции следуют одному паттерну интерфейса
- **Четкое разделение ответственности**: Основные функции и вспомогательные функции разделены

### Преимущества
1. **Читаемость кода**: Отличная документация с Doxygen комментариями
2. **Консистентность**: Единообразный стиль кодирования и именования
3. **Модульность**: Легко тестировать и поддерживать отдельные компоненты
4. **Безопасность типов**: Использование `meta_t` для метаданных
5. **Валидация**: Проверки на NULL указатели и корректность параметров
6. **Производительность**: Оптимизированные inline функции для базовых операций

### Недостатки
1. **Неполная реализация**: 
   - `s21_sub` - только заглушка
   - `s21_div` - только проверка деления на ноль
   - `s21_mul` - неполная реализация нормализации
2. **Отсутствие банковского округления**: Не реализовано в полном объеме
3. **Ограниченная обработка переполнения**: Базовая логика без детальной нормализации

### Отсутствующие функции
- Полная реализация `s21_sub`
- Полная реализация `s21_div` 
- Банковское округление
- Нормализация при переполнении
- Обработка edge cases для всех операций

## 2. Анализ ветки `feature/murk13hack/aritmetic/mult`

### Архитектурные особенности
- **Монолитная структура**: Все арифметические операции в одном файле
- **Низкоуровневая оптимизация**: Прямая работа с битами и переносами
- **Алгоритмическая сложность**: Продвинутые алгоритмы для нормализации

### Преимущества
1. **Полная реализация**: Все основные арифметические операции реализованы
2. **Банковское округление**: Корректно реализовано в `divide_and_round`
3. **Эффективная нормализация**: Алгоритм "встречного" нормализования
4. **Обработка переполнения**: Детальная логика для всех edge cases
5. **Производительность**: Оптимизированные алгоритмы умножения и деления
6. **Корректность**: Правильная обработка знаков и масштабов

### Недостатки
1. **Читаемость**: Сложный код с минимальными комментариями
2. **Модульность**: Все в одном файле, сложно тестировать отдельные части
3. **Документация**: Отсутствие Doxygen комментариев
4. **Консистентность**: Смешанные стили именования и типов
5. **Безопасность**: Недостаточная валидация входных параметров

### Отсутствующие функции
- Отдельные файлы для каждой операции
- Документация API
- Валидация входных параметров
- Единообразный стиль кодирования

## 3. План объединения решений (Логическое слияние)

### Этап 1: Архитектурная реструктуризация
1. **Создать модульную структуру** на основе `feature/arithmetic/base`
2. **Перенести алгоритмы** из `feature/murk13hack/aritmetic/mult`
3. **Сохранить единообразный API** и стиль кодирования

### Этап 2: Реализация основных функций (с корректной декомпозицией)
1. **s21_add**:
   - Строго выполнять только СЛОЖЕНИЕ мантисс. Логика вычитания из реализации Марка должна быть вынесена из `s21_add`.
   - Для разных знаков аргументов делегировать работу в `s21_sub` (edge-case), не реализуя вычитание внутри `s21_add`.
   - Сохранять модульную структуру и документацию из ветки base.
   - Взять из ветки Марка только нормализацию шкал (адаптировать), обработку переносов и «bankers rounding» через вспомогательные функции.
   - Полностью заменить макросы Марка (`GET_SIGN`, `SET_SCALE`, и т.д.) на функции из base: `_get_sign()`, `_set_sign()`, `_get_scale()`, `_set_scale()`.

2. **s21_sub**:
   - Реализовать полноценное ВЫЧИТАНИЕ мантисс с заёмами (borrow) и выбором знака результата по модулю большего операнда.
   - Поддержать нормализацию входов (выравнивание scale) и сжатие хвостовых нулей при необходимости.
   - Разрешается вызов `s21_add` только в противоположном edge-case (одинаковые знаки, но требуется сложение), однако основная логика вычитания — внутри `s21_sub`.
   - Использовать только хелперы ветки base (`_get_*`, `_set_*`) для метаданных.

3. **s21_mul**:
   - Взять алгоритм столбикового умножения мантисс из ветки Марка, оформить как вспомогательную функцию с именованием в стиле base.
   - Добавить нормализацию и банковское округление через общие хелперы; строго использовать API `_get_*`/`_set_*` вместо макросов.
   - Сохранить структуру и стиль ветки base.

4. **s21_div**:
   - Реализовать полный алгоритм деления с нормализацией, используя общие хелперы деления на 10 и банковского округления.
   - Обработать edge cases (деление на ноль, переполнение/«слишком мало»), возвращая коды ошибок как в base.
   - Исключительно функции `_get_*`/`_set_*` для работы с метаданными.

### Этап 3: Вспомогательные функции (рефакторинг под base)
1. **Перенести и переименовать из murk13hack** с приведением к стилю ветки base:
   - `compare_mantissas_96` → `_compare_mantissas_96`
   - `add_96_mantissas` → `_add_mantissas_96`
   - `multiply_mantissas` → `_multiply_mantissas_96`
   - `divide_and_round` → `_divide_by_10_with_bank_round`
   - `normalize_and_fit` → `_normalize_and_fit_192_to_96`
   - нормализация шкал «встречным» методом → `_normalize_scales_meet`

2. **Адаптировать под стиль base**:
   - Заменить все макросы Марка на `_get_sign()`, `_set_sign()`, `_get_scale()`, `_set_scale()` из `include/s21_helpers.h`.
   - Добавить Doxygen-документацию, привести сигнатуры к типам и стилю base (включая `meta_t`).
   - Разделить на логические уровни: операции с мантиссой 96-бит, операции приведения масштаба, операции округления.

### Этап 4: Банковское округление
1. **Реализовать централизованную функцию** банковского округления
2. **Интегрировать во все операции** где требуется нормализация
3. **Добавить тесты** для всех edge cases

### Этап 5: Тестирование и валидация
1. **Создать comprehensive test suite**
2. **Покрыть все edge cases**
3. **Проверить производительность**
4. **Валидировать корректность** против C# decimal

## Заключение

Объединение двух реализаций позволит создать robust, читаемый и производительный код для арифметических операций s21_decimal. Ключевые преимущества:

1. **Архитектурная чистота** из base ветки
2. **Алгоритмическая корректность** из murk13hack ветки  
3. **Полная функциональность** всех арифметических операций
4. **Правильное банковское округление**
5. **Comprehensive тестирование**

Результат будет соответствовать всем требованиям проекта и обеспечит точную эмуляцию поведения System.Decimal из C#.
